Stack Tecnológico del Proyecto Brasil
Componentes seleccionados y justificación técnica

El stack fue seleccionado priorizando tres criterios: madurez comprobada en producción, compatibilidad con normativa ecuatoriana (LOPDP, SBS) y minimización de superficie de ataque. No se usa ninguna herramienta en estado experimental.

💬 Canal de Entrada: WhatsApp Business API

  • Meta for Business · API oficial verificada
  • Webhooks sobre HTTPS/TLS 1.3
  • Autenticación: tokens Bearer + firma HMAC-SHA256
  • Sin almacenamiento de mensajes en servidores Meta por más de 30 días
  • Alternativa evaluada: Twilio (descartada por costo y latencia Ecuador)

🎙️ Speech-to-Text: Whisper / Azure Cognitive

  • OpenAI Whisper Large v3 (self-hosted en contenedor)
  • Fallback: Azure Cognitive Speech (SLA 99.9%)
  • Fine-tuning con vocabulario de seguros de salud Ecuador
  • Audio no se envía a servidores externos en modo self-hosted
  • Precisión >95% en español ecuatoriano

🧠 LLM Orquestador: Claude 3.5 / GPT-4o

  • Proveedor primario: Anthropic (Claude 3.5 Sonnet)
  • Proveedor secundario: OpenAI (GPT-4o) como fallback
  • Modo: API enterprise con DPA (Data Processing Agreement)
  • Datos de afiliados: enmascarados antes de enviarse al LLM
  • Zero data retention activado en ambos proveedores

🔗 CRM: HubSpot Enterprise

  • Integración via API v3 (OAuth 2.0)
  • Campos personalizados mapeados al schema de Saludsa
  • Acceso de solo escritura desde los agentes IA
  • Auditoría de todas las operaciones CRUD
  • Datos en servidores certificados ISO 27001

🔍 Enriquecimiento: Fuentes Públicas

  • LinkedIn API (modo lectura pública)
  • Google Custom Search para validación empresarial
  • Sin acceso a bases de datos no públicas
  • Cumplimiento estricto con LOPDP: solo datos públicamente disponibles
  • Log de cada búsqueda ejecutada y su resultado

📊 Dashboard: Metabase / Grafana

  • Metabase (self-hosted) para KPIs comerciales
  • Grafana para métricas de infraestructura y agentes
  • Autenticación: SSO integrado con directorio de Saludsa
  • RBAC: acceso diferenciado por rol (asesor / gerente / TI)
  • Exportaciones auditadas y con marca de agua

🐳 Infraestructura: Docker + Cloud

  • Todos los servicios en contenedores Docker
  • Orquestación: Docker Compose (piloto) → Kubernetes (escala)
  • Cloud: AWS São Paulo (sa-east-1) · región LATAM más cercana
  • VPC privada, sin puertos expuestos públicamente
  • Backups automáticos cada 6 horas con retención 30 días

🔐 Identidad y Acceso: Vault + IAM

  • HashiCorp Vault para gestión de secretos y API keys
  • AWS IAM con roles de mínimo privilegio
  • MFA obligatorio para todos los accesos administrativos
  • Rotación automática de credenciales cada 90 días
  • Ninguna API key almacenada en código fuente

Protocolo de Seguridad por Capas
7 capas de protección: desde el perímetro hasta el dato

El modelo de seguridad sigue el principio Defense in Depth: si una capa es comprometida, las capas siguientes mantienen la protección. Ningún componente individual puede comprometer la totalidad del sistema.

1

Capa de Perímetro · Red y Firewall

Todo el tráfico entrante pasa por AWS WAF (Web Application Firewall) y CloudFront. Solo se aceptan conexiones HTTPS/TLS 1.3. IPs de Saludsa y Vibramente en lista blanca para acceso administrativo. Rate limiting por IP: máximo 100 req/min en endpoints de agentes.

AWS WAFCloudFront CDNTLS 1.3IP AllowlistRate Limiting
2

Capa de Autenticación · Identidad y Acceso

Cada servicio y agente se autentica con tokens de corta duración (JWT, TTL 1 hora). MFA obligatorio para accesos humanos. Los agentes IA usan service accounts con permisos mínimos, auditados en tiempo real. Zero-trust: ningún servicio confía en otro por defecto.

JWT + OAuth 2.0MFAZero TrustService AccountsHashiCorp Vault
3

Capa de Aislamiento · Contenedores Docker

Cada agente IA corre en su propio contenedor aislado. Comunicación inter-contenedor solo por red interna Docker (no expuesta al exterior). Imágenes base oficiales y verificadas, escaneadas con Trivy antes de cada despliegue. Read-only filesystem donde sea posible.

Docker IsolationNetwork SegmentationTrivy ScanningRead-only FSNo Root
4

Capa de Datos · Clasificación y Enmascaramiento

Los datos se clasifican en tres niveles: Público (nombre, empresa), Confidencial (datos de salud, ingresos) y Restringido (número de cédula, datos médicos). Los datos Confidenciales y Restringidos se enmascaran (tokenización) antes de enviarse a APIs externas. Los LLMs nunca reciben cédulas ni datos médicos reales.

🔴 Restringido🟡 Confidencial🟢 PúblicoTokenizaciónData Masking
5

Capa de API · Control de Llamadas a Modelos IA

Un API Gateway centralizado (Kong / AWS API Gateway) actúa como intermediario entre los agentes y los LLMs externos. Implementa: control de tokens por sesión, filtrado de PII antes de envíos, logging de todas las llamadas, y circuit breaker automático si se detectan respuestas anómalas o intentos de prompt injection.

API GatewayPII FilterToken ControlPrompt GuardCircuit Breaker
6

Capa de Auditoría · Logs y Trazabilidad

Cada acción del sistema genera un log inmutable (append-only) en AWS CloudWatch + S3 con retención de 1 año. Los logs incluyen: qué agente actuó, qué datos procesó (referencia tokenizada), timestamp, resultado y usuario que autorizó. Imposible modificar o borrar sin dejar evidencia.

Logs InmutablesCloudWatchS3 WORM1 año retenciónSIEM
7

Capa de Cumplimiento · Regulatorio y Gobernanza

Cumplimiento con LOPDP Ecuador (Ley Orgánica de Protección de Datos Personales), regulaciones SBS, y estándares ISO 27001. DPA (Data Processing Agreements) firmados con todos los proveedores cloud. Derecho a eliminación de datos implementado en todos los módulos. Revisión trimestral de políticas.

LOPDP EcuadorSBSISO 27001DPARight to Erasure

🔄 Actualizaciones sin Downtime

Rolling updates: los contenedores se actualizan uno a uno. Si la nueva versión falla, rollback automático en <2 minutos. El sistema nunca se detiene completamente.

Herramientas IA: Uso y Políticas de Seguridad
Qué herramienta hace qué, y qué reglas gobiernan su uso

Cada herramienta de IA tiene un rol específico y delimitado. No todas las herramientas tienen acceso a todos los datos. El principio es: mínimo acceso necesario, máximo control.

🧠
Claude (Anthropic)
Agente Principal de Extracción y Cotización
Motor central de comprensión del lenguaje. Procesa transcripciones de voz para extraer datos del prospecto y genera cotizaciones personalizadas. Elegido por su capacidad de seguir instrucciones complejas y menor tasa de alucinaciones en tareas estructuradas.
✅ Permitido: Procesar transcripciones anonimizadas. Generar cotizaciones. Clasificar intención.
🚫 Prohibido: Recibir cédulas, datos médicos reales o información financiera exacta. Solo recibe datos tokenizados.
⚙️ Control de tokens: Máx. 4,000 tokens por llamada. Rate limit: 50 llamadas/hora por asesor. Zero data retention activado.
🔍
Perplexity AI
Agente de Enriquecimiento de Perfiles
Herramienta de búsqueda con IA usada por el Agente de Enriquecimiento para buscar información pública del prospecto: empresa donde trabaja, cargo, sector, presencia web. Complementa el perfil del CRM con datos verificables de fuentes abiertas.
✅ Permitido: Búsquedas de información empresarial pública. Validación de empleadores. Información sectorial.
🚫 Prohibido: Búsquedas que incluyan datos de salud, financieros privados o información familiar sensible.
⚙️ Control: Máx. 10 búsquedas por prospecto. Log completo de cada query ejecutada. Revisable por compliance.
💻
GitHub Copilot
Asistente de Desarrollo del Equipo Vibramente
Herramienta de asistencia al desarrollo de software usada exclusivamente por el equipo técnico de Vibramente durante la construcción del sistema. No tiene acceso a la infraestructura de producción ni a datos de Saludsa.
✅ Permitido: Generación de código en entornos de desarrollo. Revisión de lógica. Documentación técnica.
🚫 Prohibido: Acceso a repositorios con datos de producción de Saludsa. No se usa en ambiente de pruebas con datos reales.
⚙️ Política: Repositorios de Saludsa en instancia privada de GitHub Enterprise. Copilot deshabilitado en repos con datos sensibles.
📡
OpenAI (GPT-4o / Whisper)
STT + Agente de Respaldo
Whisper (OpenAI) se usa en modo self-hosted para transcripción de voz: el modelo corre en los propios servidores del sistema, el audio nunca sale a servidores de OpenAI. GPT-4o actúa como fallback si Claude no está disponible, con las mismas restricciones de datos.
✅ Whisper self-hosted: Audio procesado localmente. Sin envío externo de archivos de voz.
⚙️ GPT-4o (fallback): Solo activa si Claude falla. Mismo API Gateway con filtros PII. Zero retention en OpenAI Enterprise.
🚫 Prohibido: Usar la versión gratuita o personal de Whisper/GPT. Solo API Enterprise con DPA firmado.

Orquestación de Agentes IA
Cómo trabajan juntos los 4 agentes, con control y supervisión humana

La orquestación sigue un modelo pipeline supervisado: cada agente produce un output que el siguiente consume. En cualquier paso, un humano (asesor o gerente) puede intervenir, corregir o cancelar el flujo. Ningún agente opera de forma completamente autónoma sobre datos sensibles.

1

🎙️ Trigger: Voz del Asesor vía WhatsApp

El asesor envía nota de voz o texto al número de WhatsApp Business. El API Gateway recibe el webhook, valida la firma HMAC, registra el evento y pasa el audio al contenedor STT. El asesor es el único que puede iniciar un flujo.

Human-triggeredHMAC ValidationPunto de control humano
2

📝 Agente STT: Transcripción y Anonimización

Whisper (self-hosted) transcribe el audio. Inmediatamente, un módulo de PII Detection identifica y reemplaza datos sensibles (cédulas, fechas de nacimiento exactas) con tokens antes de pasar el texto a los agentes siguientes. El audio original se elimina en 5 minutos.

Self-hostedPII Auto-detectionTokenizaciónAudio TTL: 5min
3

🤖 Agente de Extracción (Claude): Datos Estructurados

Claude recibe la transcripción tokenizada y extrae: nombre, edad aproximada, composición familiar, cargo, empresa, necesidades de salud mencionadas, presupuesto estimado y señales de compra. El output es un JSON estructurado con confianza por campo. Claude solo ve texto anonimizado.

Input: texto tokenizadoOutput: JSON estructuradoConfianza por campoZero retention
4

🔍 Agente de Enriquecimiento (Perplexity): Datos Públicos

Con el nombre y empresa identificados, Perplexity busca información pública adicional. El agente opera con un presupuesto fijo de búsquedas (máx. 10) y solo agrega información verificable. Resultado: perfil enriquecido con datos laborales y sectoriales públicos.

Solo datos públicosMax 10 queriesLog completoRevisable por compliance
5

🎯 Agente de Scoring: Priorización del Prospecto

Motor de scoring multi-variable que combina los datos extraídos y enriquecidos para generar un score 0-100. Variables: perfil demográfico, señales de urgencia, capacidad adquisitiva estimada, plan más probable. Los pesos del modelo son auditables y ajustables por Saludsa.

Score 0-100Modelo auditableAjustable por Saludsa
6

📄 Agente de Cotización (Claude): Propuesta Personalizada

Claude genera la cotización con el plan recomendado, personalizada con el nombre y necesidades del prospecto. Antes del envío, el asesor recibe una vista previa en WhatsApp para aprobar o editar. El sistema puede configurarse en modo "aprobación obligatoria" o "automático" según la política de Saludsa.

✅ Revisión humana antes de envíoModo manual o autoTemplates aprobados por Saludsa
7

💾 HubSpot + Dashboard: Registro y Visibilidad

El contacto se crea/actualiza en HubSpot con todos los datos estructurados, el score, el historial de la conversación (tokenizado) y la cotización enviada. El dashboard gerencial se actualiza en tiempo real. Cada operación queda en el log de auditoría permanente.

Auto-CRMDashboard real-timeAudit log permanenteRBAC

Eficiencia de Tokens y Control de Costos IA
Cómo se optimiza el gasto en LLMs sin sacrificar calidad

El costo de los LLMs es predecible y controlable. Con los límites definidos abajo, el gasto mensual en APIs de IA es de aprox. USD 180–320/mes para 30 asesores procesando 40 prospectos cada uno. Este costo está incluido en el fee mensual de USD 1,500.

Generación de cotización personalizada Claude 3.5 Sonnet ~2,500 tokens ~800 ops ~USD 72 ✅ Bajo
Transcripción de voz (Whisper) Whisper v3 (self-hosted) N/A (audio) ~1,200 audios ~USD 0 (infraestructura) ✅ Incluido

Scoring y clasificación Algoritmo propio (no LLM) N/A ~1,200 ops ~USD 0 ✅ Sin costo API
Fallback GPT-4o (si Claude falla) GPT-4o (respaldo) ~2,000 tokens ~50 ops (est. 4% fallo) ~USD 8 ✅ Mínimo

🎚️ Límites Automáticos

Si un agente supera 150% del uso esperado en cualquier hora, se activa una alerta automática y se reduce el rate limit. El sistema no puede generar facturas sorpresa.

📊 Dashboard de Consumo

Saludsa tiene acceso a un panel de consumo de tokens en tiempo real, desglosado por asesor, agente y tipo de operación. Transparencia total.

🔄 Caching Inteligente

Respuestas repetitivas (e.g., cotizaciones para perfiles similares) se cachean en Redis por 24 horas. Reduce llamadas al LLM en ~20% sin impactar calidad.

🗂️ Clasificación de Datos del Proyecto Brasil

  • 🟢 Público: Nombre, empresa, cargo → puede enviarse a LLMs
  • 🟡 Confidencial: Edad, composición familiar, ingresos estimados → tokenizado antes de LLM
  • 🔴 Restringido: Cédula, datos médicos, historial de salud → nunca sale del perímetro

📜 Derechos del Titular (LOPDP)

  • ✅ Derecho de acceso: panel de consulta implementado
  • ✅ Derecho de rectificación: formulario en HubSpot
  • ✅ Derecho de eliminación: borrado en cascada en 72h
  • ✅ Derecho de portabilidad: exportación JSON/CSV
  • ✅ Consentimiento explícito antes del primer procesamiento

🔒 Cifrado en Reposo y en Tránsito

  • Base de datos PostgreSQL: AES-256 en reposo
  • S3 (logs, backups): SSE-S3 con claves propias
  • Todo tráfico: TLS 1.3 con Perfect Forward Secrecy
  • Claves gestionadas en HashiCorp Vault (no en código)
  • Rotación de claves: cada 90 días automático

🚨 Plan de Respuesta a Incidentes

  • Detección automática: SIEM + alertas en <5 min
  • Contención: aislamiento de contenedor comprometido en <2 min
  • Notificación a Saludsa: en <1 hora hábil
  • Notificación SBS/ARCOTEL: en <72 horas (cumplimiento LOPDP)
  • Forensics y reporte post-incidente: en 5 días hábiles

Resumen para el Directorio de Seguridad Digital
Los 6 puntos clave para la aprobación del proyecto

1️⃣ Los datos sensibles nunca salen del perímetro

Cédulas, datos médicos e historiales de salud son restringidos: no se envían a ningún LLM externo. Solo los datos tokenizados o de categoría pública viajan a APIs de IA.

2️⃣ Arquitectura de defensa en profundidad (7 capas)

Un atacante que supere una capa enfrenta 6 más. Cada componente está aislado en Docker. No hay un único punto de fallo que comprometa todo el sistema.

3️⃣ Cumplimiento regulatorio nativo

LOPDP, SBS, ISO 27001. DPAs firmados con todos los proveedores cloud. Derecho a eliminación implementado. El sistema está diseñado para pasar una auditoría regulatoria.

4️⃣ Control humano en cada paso crítico

Los agentes IA no operan de forma autónoma con datos sensibles. El asesor aprueba la cotización antes del envío. Los gerentes tienen visibilidad total en el dashboard.

5️⃣ Costo de IA predecible y transparente

~USD 180–320/mes para todo el equipo comercial. Con límites automáticos, caching y dashboard de consumo en tiempo real. Sin sorpresas en la factura.

6️⃣ Auditoría completa y trazabilidad

Cada acción queda en logs inmutables por 1 año. Si algo sale mal, sabemos exactamente qué agente hizo qué, con qué datos y en qué momento. Trazabilidad forense completa.
